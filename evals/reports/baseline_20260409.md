## 横断的な弱点パターン

### A. Planner JSON出力の脆さ (発生率 ~20%)

| 失敗トリガー | 例 |
|---|---|
| 空/曖昧入力 | `j3`, `ec1` |
| 長い code 文字列 (改行/quote 含む) | `m1`, `m4` |
| replan時 (履歴付き再生成) | `m4` |

**FT待たず即対応:**
- Plannerプロンプトに「`code` フィールドの改行は `\n` でエスケープ」「JSONのみを出力」「迷ったら最小ステップ」を強化
- `_extract_json` の brace-balancing は既に実装済みだが、`\` 単独 escape の救済が未実装。`json.loads(strict=False)` も限界がある
- 空入力時は Planner を呼ばず「タスクが空です」エラーを返す前段ガード追加

### B. ハルシネーション (発生率 ~25%)

| ケース | 例 |
|---|---|
| 知らない固有名詞を即答 | k4 (架空イベント) |
| バグがないコードにバグを発見 | c3 (Yes-bias) |
| 論理的に確実でないものを断言 | r1 (推移律の誤適用) |
| 自己認識を盛る | k3 (Gemini と誤答) |

**FTで学ばせるべきパターン:**
- 「知らない」と言える訓練データ
- 「指摘事項なし」と返せる訓練データ
- 自己同一性 (Gemma 3 12B / Ollama 経由 / ローカル動作) を明示する SFT データ

### C. 安全性のゼロ実装 (発生率 100%)

`rm -rf` を平然と生成。sandbox 未実装。**FT 以前に Planner プロンプトとガード層を入れるべき**。

### D. 出力長の途中切れ

ハーネスで `output[:1500]` で打ち切っているのも一因だが、`code_executor` で長いコードを返す場合 Ollama デフォルトの `num_predict` (128 トークン?) で実質切れている可能性。`options.num_predict=2048` を設定すべき。

### E. ツール間データ受け渡しの誤理解

`j2` で `echo $LAST_COMMAND_RESULT` を生成。Planner はステップ間で結果が引き継がれる仕組みを把握していない。これは Planner プロンプトに「ステップ間のデータ受け渡しは Python の code_executor 内で完結させる」と明記すべき。

## FT に向けた推奨アクション

### 即時対応 (FT前にやるべき)

1. **Plannerシステムプロンプト強化** (`src/agent/core/planner.py`)
   - 安全性 (破壊的コマンドの拒否)
   - JSON出力ルール (escape, 単発出力)
   - ステップ間データ受け渡しの説明
   - 「不明確な指示は最小限の調査ステップに留める」

2. **`OllamaClient` の `num_predict` 引き上げ**
   - デフォルト 4096 など (現状未指定)

3. **空入力ガード** in `Planner.plan()`

4. **JSON抽出ロジック改善** — `\` escape の救済

### FT (QLoRA) で学ばせるべき

| 学習目標 | 必要データ件数 (目安) | 入手方法 |
|---|---|---|
| 「知らない」と言える | 200〜500 | 既存QAデータセット (TruthfulQA等) + 自作の架空質問 |
| 「指摘事項なし」と言える | 100〜300 | 自作 (バグなしコード + "問題なし"応答) |
| Plannerの構造化出力固定 | 500〜1000 | 自作 (本プロジェクトの実エピソードを学習データ化) |
| 自己同一性 | 50〜100 | システムプロンプト + SFT |
| 安全な拒否 | 200〜500 | 既存 redteam データセット流用 |

### 学習データの自動生成パイプライン (Phase 4で実装)

`AgentSession` のエピソード記憶 (ChromaDB) には既に成功/失敗履歴が貯まっている。これを:
- 成功エピソード → そのまま正解データ
- 失敗エピソード → 人手 or LLM-as-judge で修正した上で正解化
- 「Plannerが正しい構造化出力を出した」サンプルを集中的にDPO学習

## 結論

**現状の Gemma 3 12B + 自作エージェント** は、

- ✅ 通常のコード生成・ファイル操作・標準的な質問応答は実用レベル
- ⚠️ Planner JSON出力の安定性が ~80% で、商用には足りない
- 🔴 安全性 (sandbox / 拒否) は **未実装**
- 🔴 ハルシネーション抑制が弱く、「知らない」と言えない

**FTのスコープ提案**: 上記「FTで学ばせるべき5項目」を `unsloth + QLoRA` で順次実施。データは既存OSSデータセット流用 + 本リポジトリの実エピソードから自動生成のハイブリッドが最効率。

ただしまずは**FT前に Plannerプロンプト強化と num_predict 設定** を行うだけで体感的に大幅改善する見込み。**安全層 (PathGuard / コマンドフィルタ) は最優先**。
